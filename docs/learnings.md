# Symbion — learnings

Accumulated patterns from each feature's Maker→Checker→QA loop. Confidence levels: **high** (proven across multiple reviews/features), **medium** (worked once, worth repeating), **low** (worked but circumstantial, re-verify next time).

## Pipeline process

- **A path-confinement guard fix that adds a new unconditional check (`rejectTraversalSegments` moved to the top of `resolveConfinedPath`) can be 100% correct in shipped code while every existing negative test still passes against the *unfixed* code — because the existing tests' attack strings happened to start with `..`, which coincidentally satisfies an older, weaker string-prefix check too.** Security review caught this via live mutation testing (temporarily removing the new call, reconfirming all 229 tests still passed, then constructing a non-`..`-prefixed Windows-style traversal string that the old code would silently allow). The fix: add a negative test whose attack string does NOT start with `..` (e.g. `"subdir\\..\\..\\escape.md"`) specifically so the test can only pass by exercising the real segment-membership check, not a string-prefix coincidence. General principle: when a security fix changes *how* a rejection is detected (not just adding a new rejection), audit whether existing tests would coincidentally still pass against the *old, broken* logic — "tests pass" and "tests prove the fix works" are different claims. — **confidence: high** (caught a real CI blind spot, verified by reverting and re-running).
- **Running a full pipeline (analyze→think→design→plan→build→review→cso→qa→ship) unattended via a cron-triggered autopilot, with zero human present to answer taste questions in real time, worked end-to-end by always picking the minimal-scope/most-reversible/cheapest-to-undo option at every open question and explicitly documenting each pick (with rationale) in STATE rather than silently guessing.** Applied at THINK (Windows-path scope chosen as "fix path-string validation only, no native Windows packaging" — the smaller of two very different-sized readings of the same issue text) and at DESIGN (placeholder copy chose platform-neutral over Windows-flavored, declined to add an unrequested content addition to a confirmation message). None of these decisions touched already-shipped behavior; all are one-line-reversible if a human reviewer disagrees. This is the corrected pattern the "RETRACTED" entry below warns against repeating incorrectly: the key difference is treating every guess as a *documented, reversible default* flagged for review, not a silent substitution presented as if no decision were made. — **confidence: medium** (worked once, end-to-end, no human intervention needed; re-verify this generalizes to a feature where the minimal-scope reading and the literal issue text genuinely conflict, not just differ in size).
- **A second consecutive unattended autopilot run (`load-existing-workflows`, Issue #8) confirmed the "minimal-scope default + document for review" pattern generalizes — and also confirmed the value of having a BA agent read the actual code before writing the spec.** The issue's literal text ("Symbion only supports creating new workflows") implied a missing-parser feature; reading `packages/core/src/parse/scan.ts` and the daemon's `scanClaudeDir`/`importArtifacts` handlers up front revealed the parser/import RPC pipeline already existed and was already read-only-safe — the actual bug was a pure UX/detection gap (`CreateProjectDialog` showing a passive, easy-to-miss hint instead of a real interrupt). This downgraded the feature from "new provider-adapter parsing work" to "a pure `apps/web` UI-wiring change," which in turn meant `/cso` was correctly skipped (verified via `git diff --stat` against `apps/daemon`/`packages/core`/`packages/rpc-types` showing zero touches) — the size of a feature should be established by reading the code, not by taking the issue's framing at face value. — **confidence: high** (the "is this actually a UX gap or a missing-capability gap" question determined the entire shape of the loop, and was answered correctly by code-reading before any design/plan work started).
- **An independent code-reviewer catching a real 🔴 (not just a style nit) on a "PASS"-bound feature, on the second-to-last gate before QA, confirms the dual-review step still earns its cost even on small, low-risk-looking UI-wiring features.** The bug — `currentProject` never refreshed with `importArtifacts`'s merged result, so the dialog closed onto a project view showing zero artifacts despite a successful server-side import — existed in BOTH the new code (`CreateProjectDialog`) and a pre-existing call site (`ImportDialog`) that the feature's diff happened to duplicate into a second location. Architect's independent pass on the same diff did not catch this (it reviewed structure/drift, not the specific data-flow path from RPC response to rendered UI) — reinforcing that code-reviewer and architect roles are genuinely complementary, not redundant, even when both return PASS-leaning verdicts. The fix was applied via a narrow scoped re-review (not a full re-review), consistent with the established "narrow re-review closes a single remaining gap" pattern. — **confidence: high** (a real user-visible bug, caught by exactly the review step designed to catch it, fixed and re-verified before ship).

- **Independent dual-review (code-reviewer + architect in parallel) catches different bug classes than either alone.** On the v1 scaffold, code-reviewer found the E9 heartbeat gap and a real semver string-comparison bug; architect independently flagged the missing Playwright e2e and the hand-mirrored `@symbion/rpc-types` as the most consequential architectural risk. Neither fully overlapped. — **confidence: high**
- **Reviewers that re-run the build/test suite themselves (not just read the diff) catch self-report drift.** Round 2's architect did a clean `rm -rf dist && npm run build` to prove the workspace build order resolved without manual path hacks, rather than trusting the Maker's claim. Round 3 caught that an e2e spec's *header comment* claimed S8 coverage it didn't actually have — only found by reading the spec body, not the Maker's summary. — **confidence: high**
- **A narrow, scoped re-review (not a full re-review) is the right tool for closing a single remaining gap.** After round 2 left exactly one open item (S8 e2e coverage), round 3 was scoped to just that item plus a deviation the Maker flagged (a bug fix outside the stated scope) — fast, and didn't re-litigate already-confirmed-fixed items. — **confidence: medium**
- **When a Maker finds a real bug while doing unrelated work (e.g. fixing a test-coverage gap), having it explicitly flag the deviation rather than silently fixing it lets the Checker scope its re-verification correctly.** The `WorkflowForm.tsx` stale-closure bug (found while adding S8 e2e coverage) was called out by name and re-verified narrowly instead of triggering a full re-review. — **confidence: medium**

- **An agent that reports "failed" due to hitting an API session limit may still have completed and persisted all of its work before the error — check the actual file state before assuming a partial/broken result.** `boot-terminal-ux`'s `/build` run showed a `status: failed` notification with the session-limit error, but the STATE file's `## BUILD` section ended cleanly (not mid-sentence) and every file/test it claimed to have written actually existed and passed when independently re-run (`menu.test.ts` unchanged 7/7, new `banner.test.ts`/`openBrowser.test.ts`/heartbeat/badge tests all green). The failure happened at the tail end of the agent's own wrap-up, after the real work was already saved. General rule: on a "failed" subagent notification, read the artifact it was writing to and independently re-verify (re-run tests, don't just trust the last text) before deciding whether to redo the work — don't assume "failed" means "nothing happened." — **confidence: medium** (one confirmed instance; the failure mode — dying after the write but before the final chat message — may not generalize to every kind of mid-task API interruption).
- **When a user explicitly overrides the `/ship` gate (skips `/review`+`/qa`+`/cso` and asks to ship anyway) after being told the concrete risk, the correct response is to record the skip and residual risk in STATE before committing, not to refuse or to silently ship without documenting it.** `/ship`'s own precondition instructions require this ("if it isn't recorded, stop and ask... then write it down before continuing") — asking once via `AskUserQuestion` with the risk spelled out, then writing a dedicated `## SHIP — gate status` section naming exactly what wasn't verified (no independent Checker pass, no live QA including the deferred manual terminal checks and e2e suite, no security audit of a token/session-auth change) gives a future reader/Checker a precise starting point if something breaks post-ship, instead of a silent gap. — **confidence: medium** (first time this exact override path was exercised end-to-end in this project's pipeline).

## Architecture / build

- **A "shared type package" requirement in a plan (e.g. STATE §8) should be implemented as a literal third workspace package, not a hand-mirrored file with a "keep these in sync" comment.** The first build attempt took the comment-based shortcut; both reviewers flagged it as the single biggest architectural risk even though it happened to be in sync at review time — "currently correct" isn't the same as "structurally enforced." Fix was to add `packages/rpc-types` as a real type-only workspace package that both `apps/daemon` and `apps/web` import from. — **confidence: high**
- **A pure `packages/core` (no fs/net/Node imports) is enforceable and verifiable by grep** (`node:`, `fs`, `path`, `child_process`, `process.`, `__dirname`) — cheap to check every review round, caught zero violations across 3 rounds once the initial scaffold was built correctly. — **confidence: high**
- **Path-confinement guards need a two-layer test**: (1) a relPath-whitelist filter before the guard even runs (defense in depth — paths that don't match any known artifact are dropped before reaching disk logic), and (2) the guard itself (`resolveConfinedPath`) rejecting `..`-escapes, absolute paths, and symlink-escapes via `realpathSync` ancestor walk. Both layers were exercised live in QA and held. — **confidence: high**
- **React stale-closure bugs in form "insert at cursor"-style helpers** (two sequential `update()` calls against the same closed-over object, where the second clobbers the first) are a recurring risk in any builder UI with helper buttons (`[Chèn $ARGUMENTS]`-style inserts). Single merged `onChange({...artifact, ...changes})` calls avoid it. Watch for this pattern in any future multi-field form helper. — **confidence: medium**

## Testing

- **Daemon-process e2e tests (Playwright spawning the real built daemon binary, not a mock) are worth the setup cost** — they caught the stale-closure form bug that unit tests entirely missed (unit tests covered the IR/render logic, not actual button-click behavior). — **confidence: high**
- **When chrome-devtools/Chrome isn't available in the sandbox for manual QA, driving the same production daemon directly over its real RPC contract (matching headers/auth/Origin exactly as the web client would) is a valid substitute** for verifying backend behavior — it exercises the identical code path the UI calls, just skips the browser-rendering layer. Not a full substitute for verifying UI-level behavior (a11y, visual state, button-disabled-in-DOM), so still flag that gap explicitly rather than silently treating RPC-level testing as equivalent. — **confidence: medium**

## Office-hours / scope

- **RETRACTED (see corrected pattern below) — "resolve a locked-decision conflict by silently reinterpreting the ask" was the wrong call, not a validated pattern.** A prior version of this entry praised full-autopilot's choice to satisfy issue #2's "auto-generate using local AI" by shipping a deterministic string-template heuristic with zero LLM/network involvement, framed as "a reinterpretation that satisfies the user-visible need without reopening the lock." The user's actual follow-up feedback (`docs/loops/auto-generate-body-STATE.md` §0) rejected this outright: office-hours had *already correctly identified* "which AI backend" as the single most important blocking decision requiring the user's literal pick, then overrode its own self-identified blocker by inventing and shipping an option nobody asked for, under autopilot, with no sign-off. That is a process failure, not a successful reinterpretation.
- **When office-hours/autopilot self-identifies an open question as "the single most important blocker requiring the user's pick," that determination must be honored — autopilot may not then silently substitute its own answer to keep shipping, even if the substitute is technically simpler/safer/zero-new-dependency.** The tell that this failure is occurring: the same phase, in the same document, writes both "this blocks architecture design, please pick" AND a few paragraphs later picks something itself anyway, often re-citing an unrelated already-locked constraint to justify the substitution. If a locked decision and a new feature's literal ask conflict, the correct move under autopilot with no user present is to **leave that item open and proceed only on the unambiguous parts**, not to resolve it with a guess and ship the whole feature. The cost of being wrong here is high: it produced two shipped-then-reverted PRs on the same issue. — **confidence: high** (learned from a confirmed, user-flagged real-world failure, not a hypothetical).
- **A pure-function "AI-flavored" feature (heuristic text generation with no network/model call) still needs the same never-throws and output-format-safety rigor as a real backend integration.** Code review caught a throw-on-null-field bug and a string-corruption bug (truncation through a parenthetical clause) in the heuristic's edge-case handling — these are the same class of bug a real LLM-backed feature would have for malformed responses, just triggered by malformed *input* instead of malformed *output*. Don't relax the rigor of review just because "it's just string templating, not a real model call." — **confidence: medium**
- **A locked STATE doc that writes out its own root-cause verdict and then gates every open question behind an explicit "user must answer, no autopilot default" instruction successfully prevented the same failure from recurring.** On the corrected feature (`auto-generate-body-STATE.md`), 12 open questions in §5 were surfaced via `AskUserQuestion` instead of guessed, including the central provider/key-custody decision (resolved as "both, pluggable, Ollama default" — neither autopilot's original heuristic-substitute nor a single forced remote-API pick). Writing the failure mode explicitly into the spec, not just fixing the immediate bug, is what made the second attempt behave differently. — **confidence: high**

## Real-AI integration (daemon-side LLM calls)

- **The first network-egress-capable RPC method in a previously filesystem-only daemon is a genuinely new trust boundary and should be treated as such by `/cso`, not folded into a routine code review.** This feature's security review (independently, via live mutation testing — reintroducing each bug and confirming the regression test actually fails) caught a real 🟠 HIGH SSRF: an env var (`SYMBION_OLLAMA_BASE_URL`) added by the Maker without a locked design slot for it, read with zero validation that the resolved host was loopback. Any new "Maker added an undocumented env var for test-injection convenience" deserves an explicit SSRF/host-allowlist check whenever that var can influence an outbound `fetch` target. — **confidence: high**
- **Provider/adapter interfaces (`LlmProvider` with Ollama + remote-stub adapters behind a registry) are worth building even when only one adapter is reachable from the UI in v1** — it gave the architecture review a concrete seam to verify ("is the remote path a real seam or just a TODO comment") and meant the inevitable next-feature ask ("now wire up the remote provider's UI") is additive, not a rewrite. — **confidence: medium**
- **A "no dynamic model-list fetch" office-hours answer is compatible with adding a `listModels` RPC, as long as the RPC itself makes zero network calls (just echoes the daemon's own hardcoded list).** The architect initially flagged the resulting "model list duplicated by hand in web + daemon" as a real risk (R1) rather than silently picking a fix; surfacing it back to the user for a one-question decision (add the RPC vs. accept the duplication + a tripwire test) resolved it correctly instead of guessing which reading of "no dynamic fetch" was intended. — **confidence: medium**
- **Mutation testing during security re-review (deliberately reintroduce the reported bug, confirm the new regression test fails, then restore and confirm it passes) is a meaningfully stronger verification step than reading the diff and trusting the Maker's fix description** — it caught that a plumbing detail (moving `getProvider()` inside a `try` block so a new constructor-time error routes through the existing `RpcError` mapping) was subtle enough to be worth proving, not assuming. — **confidence: high**

## Full-app dark redesign port (Symbion v2 — left-rail, DESIGN.md tokens) — 2026-07-08

- **A build agent's own completion signal can be flagged as unreliable ("safety classifier unavailable... verify before acting on it") and skip its own required self-report — treat that as a hard trigger to independently re-verify before trusting anything downstream, not just a formality to note and move past.** On this feature, the build agent's task notification carried exactly that warning and never wrote the STATE "BUILD notes" section it was told to write. Rather than proceeding to `/review` on faith, the orchestrator ran its own spot-checks first (`npm run build`, `git diff --stat -- packages/core apps/daemon`, reading the actual store diff) and wrote the BUILD notes itself from those findings — explicitly labeled as independently-verified, not a transcript of the agent's claims. That extra step is what caught the framing gap before it reached the Checker. — **confidence: high** (a real, non-hypothetical trust-boundary signal from the harness itself, acted on rather than dismissed).
- **Even after that extra caution, both independent reviewers (code-reviewer + architect, running in parallel with no coordination) still found a 🔴 blocker the orchestrator's own spot-check missed: an entire planned screen (the Graph tab) was silently skipped, with orphaned-but-unwired component files making it look superficially done from a `git status` glance alone.** The orchestrator's spot-check confirmed *files existed* and *the build compiled*; it never grepped whether the new components were actually referenced inside the JSX that would render them. Lesson: "the file exists" and "the file is wired into the render tree" are different claims, and only a diff-level read of the actual consuming file (not just `git status`/`ls`) catches the difference — this is the same class of gap as the RPC-response-not-consumed bug from an earlier feature (see "An independent code-reviewer catching a real 🔴..." entry above), just at the component-wiring layer instead of the data-flow layer. — **confidence: high**.
- **When a re-review is needed after a targeted fix-pass, explicitly instruct the reviewer to distrust the fix-pass's own self-report and re-derive each claim from the diff — reusing the *exact* verification steps that caught the original gap (e.g. "grep for the prop name inside the JSX, not just the import statement").** Both reviewers' re-review prompts named the precise prior failure mode ("declared-and-unused" vs. "actually referenced") and both independently re-confirmed it was fixed via the same grep-based method, rather than accepting the fix-pass's "confirmed wired" claim at face value. This closed the loop without a third round. — **confidence: high**.
- **Splitting a tangled working tree into (a) an unrelated-but-already-reviewed change (commit alone, on its original branch) and (b) the actual feature (moved to its own dedicated branch, checkpoint-committed before sending a fix-pass) turned an architect-flagged "one undifferentiated uncommitted blob on the wrong branch" complaint into a genuinely bisectable two-commit history, and both reviewers explicitly confirmed the process gap was closed in the re-review.** This is worth doing as a matter of course whenever a build agent leaves multiple logically-distinct diffs uncommitted in the same working tree — don't wait for a reviewer to complain about it before separating concerns. — **confidence: medium** (worked once here; the "commit a checkpoint before sending a fix-pass back" half of this pattern is the more broadly reusable piece).
- **When a design doc explicitly marks a prop/data-contract as "TBD — see Open Q N.N" and the later PLAN phase resolves that question with a different, more specific shape than the design doc's speculative sketch, following PLAN's resolution is correct, not architectural drift** — the architect reviewer explicitly ruled on this distinction (PLAN is later-in-time and authoritative once a design-doc placeholder was never locked) rather than mechanically diffing final code against the design doc's earlier draft. Worth remembering when synthesizing multi-pass design docs: mark genuinely-undecided contracts as "TBD" rather than a confident-looking guess, so a later PLAN resolution isn't misread as a deviation. — **confidence: medium**.
- **A "no browser available in this sandbox" gap during QA should be stated as an explicit, named QA finding ("mechanical PASS, visual UNVERIFIED") rather than silently omitted or silently treated as equivalent to a full pass** — for a feature whose entire deliverable is visual (a UI redesign), skipping this disclosure would misrepresent what was actually checked. Mechanical checks (build, route 200s, token values at the source-config level) are real signal but are not a substitute for seeing the rendered page, especially for a feature explicitly graded on pixel/token fidelity. — **confidence: high** (directly extends the "canary/verify" honesty principle to verification claims about a feature's core deliverable, not just to filesystem writes).

## Investigate: Create Project — "không tạo được dự án với đường dẫn" (2026-06-27)

- **Root cause**: `CreateProjectDialog.tsx`'s `canCreate` gate requires `validation.exists && validation.isDir` — by design (`symbion-STATE.md` §453: "create project" only *registers* an existing repo path, v1 never creates folders). The bug report is the expected behavior for a not-yet-existing path, but it's compounded by two real gaps against the locked design: (1) `symbion-design.md:352` and `symbion-testplan.md:97` spec a `[Chọn…]` browse-folder button in the dialog that was **never implemented in the web UI** — `CreateProjectDialog.tsx` has no button wired to the `browseFolder` RPC at all; (2) the `browseFolder` RPC itself (`apps/daemon/src/fs/folderPick.ts`) is an intentional permanent stub that always returns `{cancelled:true}` (documented reason: no native Electron/Tauri dialog dependency wired in), so even if the button existed, v1 had no working folder-picker path — only the typed-path fallback, with no escape hatch when the typed path doesn't exist yet.
- **Why this was hard to spot from the UI alone**: the dialog does show `✗ Thư mục không tồn tại`, but gives no actionable next step (no create-folder option, no folder browser), so the failure reads as "create project is broken" rather than "this is the documented existing-folder-only v1 behavior missing its companion browse affordance."
- **Fix direction chosen** (routed to `/plan` → `/build`, not fixed inline): (a) replace the dead `browseFolder` native-dialog stub with a daemon-backed directory listing RPC + in-app browser modal (no native dialog lib needed, works through the existing localhost-RPC + path-confinement model); (b) add a "create this folder?" affordance when `validatePath` reports `exists:false`, gated through the same backup/confine/atomic-write discipline as other daemon fs mutations. — **confidence: high** (root cause confirmed by reading the dialog gate, the RPC stub's own doc comment, and the locked design/testplan docs — not guessed).

## Investigate: "chỉ click xem được geochat" — rail select silently no-ops for ghost projects (2026-07-09)

- **Reported symptom vs. real cause**: user saw "only geochat opens when I click it in the rail" and read it as a *rail click* bug. It was not — every rail click calls `loadProject(id)` correctly. The real cause was **`loadProject` RPC failing with a raw `ENOENT`** for two projects whose registry entry in `~/.config/symbion/config.json` pointed at a folder that no longer had `.symbion/store.json`: `vpo` (folder still exists, is a real repo, but its `.symbion/` was emptied — `store.json` gone) and `qa-test-project` (its root `/tmp/.../qa8/fixture-claude` — a leftover /qa fixture that leaked into the *real* config — was cleaned up entirely). geochat still had its store.json, so only it opened. Confirmed by driving the daemon's `loadProject` RPC directly for all three ids (geochat → full project JSON; vpo/qa-test → `ENOENT ... /.symbion/store.json`) plus `ls -la` on each folder.
- **Why it looked like "nothing happens" (the silent-failure layer)**: `AppShell.tsx`'s `onSelectProject={(id) => loadProject(id)}` has **no `.catch`**, so a rejected `loadProject` promise becomes an unhandled rejection, `currentProject` is never set, and there is **no toast** — the click just appears dead. (Contrast: the mount-time `loadProject(openProjectId).catch(...)` at AppShell.tsx:41 does swallow errors, but at least doesn't dangle.)
- **The deadlock this created for the just-shipped remove-project feature**: the new "Remove project" button lives *inside ProjectView*, which only renders once a project is loaded (`currentProject != null`). A ghost project can never be loaded → ProjectView never renders → **the only UI affordance to forget the ghost is unreachable**. So a project whose store went missing is both un-openable AND un-removable through the UI. This is the real design gap worth fixing, not just the missing store files.
- **Fix direction chosen** (routed `/build` → `/review` → `/qa`, per Iron Law — not fixed inline): (1) web catches `loadProject` failure and shows an actionable toast ("Project folder or store missing — it may have been moved or deleted"); (2) make ghost projects removable from the rail directly (not only from inside ProjectView), so a project you can't open can still be forgotten; (3) daemon `loadProject` returns a meaningful error code (e.g. `project-missing`) instead of a raw `ENOENT`; (4) data cleanup: remove both `vpo` and `qa-test-project` from `config.json` (user chose "forget from list" for vpo rather than re-init). — **confidence: high** (root cause confirmed by direct RPC probes + on-disk inspection, not guessed; the silent-no-op and the un-removable-ghost deadlock both verified by reading AppShell/ProjectView wiring).
- **General principle**: when a user describes a symptom as "control X doesn't work," verify whether X's handler actually ran and *failed silently* before assuming X is wired wrong — an unhandled promise rejection in an event handler presents identically to "the handler never fired." And: any list whose rows open a detail view should let you *remove* a row without first successfully *opening* it, or a broken row becomes permanently stuck.

## Provider connect/setup UX (Ollama reachability check) — 2026-06-27

- **A "guide-only" status-check feature (no daemon-spawned install/pull) is enough to turn a dead-end error into a self-service fix, without expanding the daemon's trust boundary beyond read-only loopback HTTP.** The whole feature was: one new RPC (`checkProviderStatus`) doing a lightweight `GET` with a timeout, a pure OS-detection/install-command lookup table, and two small web components reusing the existing `Dialog` primitive. No new shadcn component, no polling, no settings-page infrastructure. Confirms the spec's own instinct (§8 risk notes) that the "delightful" auto-install version is exactly the version that should be deferred pending its own security review. — **confidence: high**
- **Extracting shared guarded logic (here: `resolveOllamaBaseUrl`'s SSRF/loopback check) out of a single call site into an exported function, reused by both the original caller and a new caller, is the correct way to add a second consumer of security-sensitive logic — never reimplement the guard.** Both reviewers independently verified the extraction was byte-for-byte behavior-preserving via regression tests (TC-D10/TC-D11) rather than trusting the diff. — **confidence: high**
- **A shared UI primitive (`Dialog`) silently missing a documented behavior (Esc-to-close) only surfaced because the testplan had an explicit manual checklist item (M6) for it — code review's automated checks (grep, type-check, unit tests) had no way to catch a missing keyboard handler.** Worth remembering: keyboard-accessibility behaviors need an explicit testplan line item, since they're invisible to grep-based review and untestable without a DOM/RTL harness (which this repo's web app doesn't have). — **confidence: medium**
- **A single shared boolean UI-feedback state (e.g. `copied`) breaks the moment a component renders more than one instance of the control it's meant to flag (e.g. multiple copy buttons in a multi-variant fallback view).** This bug only existed in the rarely-exercised "low-confidence/unknown-OS" branch — exactly the branch flagged by the Maker as untested due to no component-test harness. When a coverage gap is explicitly flagged at build time, it's worth a reviewer spending extra attention on that exact code path rather than skimming it, since "no test" and "no review attention" compounding is how these slip through. — **confidence: medium**

## Investigate: Generate Body 404 loop — "model not found" swallowed into a generic error (2026-06-28)

- **Root cause**: the user's local Ollama (v0.30.11, confirmed reachable, `/api/version` responds) has **zero models pulled** (`/api/tags` → `{"models":[]}`). `ollamaProvider.ts`'s 3 hardcoded model tags (`llama3.2:1b`/`llama3.1:8b`/`llama3.1:70b`) are placeholders never validated against what's actually downloaded on a given machine (already flagged as a "dev-time content decision" in the file's own header comment). Selecting any of them and clicking Generate sends a real `POST /api/generate`, which Ollama correctly 404s with `model 'llama3.1:8b' not found`. Reproduced directly via curl against the real local Ollama instance — not guessed from the screenshot's `[GIN]` log alone (that log format is Ollama's own Gin-framework access log, not Symbion's daemon log — easy to misread as "Symbion's server returned 404" when it's actually "Ollama's server returned 404 to Symbion's request").
- **A second, compounding bug found while tracing the path to the UI**: `ollamaProvider.ts` already throws a *specific* `LlmError("invalid-response", "Mô hình \"llama3.1:8b\" không tồn tại trên Ollama (404).")`, but `handlers.ts`'s `humanMessageForLlmError(code)` (the daemon→RPC error-message boundary) re-derives the user-facing message **purely from the error code**, discarding `err.message` entirely. Every `invalid-response` case — "model not pulled" and "malformed JSON from a different failure" alike — collapses to the same generic `"Phản hồi không hợp lệ từ mô hình."` ("Invalid response from the model"), which gives the user zero actionable signal (no mention of which model, no mention of `ollama pull`).
- **Lesson**: when a `code`+`message`-carrying error crosses a process/RPC boundary, re-deriving the user-facing string from the code alone (instead of threading the original message through) silently destroys any context-specific detail the throw site bothered to construct — even when that detail was specifically written to be actionable. Worth checking, in any future error-taxonomy boundary, whether the "human message" mapping function takes the *whole error* or just its discriminant. — **confidence: high** (root cause reproduced directly via curl against the real daemon/Ollama, not inferred from logs alone; the message-discarding bug confirmed by reading both `ollamaProvider.ts` and `handlers.ts` and tracing the exact call site).

## self-coded-graph-migration — a jsdom-invisible SVG-namespace bug proved why "no reachable browser" residual risk isn't uniform severity across features (2026-07-17)

- **`<div>` elements nested inside `<svg>`/`<g>` JSX are created in the SVG XML namespace by React, not the HTML namespace — they will not receive `position: absolute`, CSS classes, or normal box-model layout in a real browser, and jsdom-based tests cannot detect this class of bug unless they specifically assert `namespaceURI`.** A self-coded graph-edge component's badge/toolbar/delete-confirm `<div>` was rendered as a JSX child of `<svg><g>` (replacing xyflow's `EdgeLabelRenderer`, which portals such content out of the SVG tree for exactly this reason) — the component's own tests, and every other test in the migration, passed cleanly because jsdom's `querySelector`/`getByText` still "find" an SVG-namespaced div in the tree even though it's invisible/unstyled in a real browser. Two independent Checkers (`code-reviewer`, `architect`) caught this via careful reading of the DOM content model, not via any test failure — `code-reviewer` then empirically reproduced the failure with a throwaway RTL test asserting `namespaceURI` before trusting the finding. **Lesson: for any self-coded SVG/HTML-mixed rendering surface, "renders in jsdom without error" is not evidence of DOM-content-model validity — the specific check needed (does this element's ancestor chain up to `<svg>` contain only SVG-valid elements, or does an HTML element leak in) requires either a real browser or an explicit `namespaceURI`/structural-source assertion, and won't be caught by generic snapshot/existence-based testing.** — **confidence: high** (empirically reproduced by 2 independent methods: `code-reviewer`'s throwaway RTL repro, and later the Maker's own fix-pass repro before trusting its regression tests — both confirmed the exact same failure signature).
- **When a Checker finds and self-critiques a gap in its own prior plan (here: `architect`'s §9.1.1 never specified a test to guard the exact invariant its own text described), the fix pass closing that gap should be verified with a method independent of "reading the new code" — ideally by reconstructing the ORIGINAL bug and confirming the new test actually fails against it.** In round 2, `code-reviewer` didn't just read the fix and the new regression tests — it reconstructed the pre-fix broken structure in a scratch copy, ran the new tests against it (3 of 5 assertions failed, reproducing the exact original error message), then restored the fix and re-ran (all pass). This is meaningfully stronger evidence than "the test exists and currently passes," since a test can exist, look plausible, and still be accidentally vacuous (e.g., testing the wrong element, or an assertion that would pass regardless of the bug). **This "revert and reproduce" verification pattern — first seen in `graph-execution-realtime` P3's security re-review, now confirmed a second time in an unrelated feature by a different Checker role — is worth treating as a standing best practice for any round-2 review of a fix for a Checker-found defect, not situational.** — **confidence: high** (second independent occurrence of the exact same verification technique, by different Checker roles, in different features, within the same session).
- **The severity of an unverified-live-browser residual risk is not uniform across features — it should be explicitly weighted by (a) whether the change is additive (new UI layered on working code) vs. wholesale-replacement (the entire rendering path for a primary surface), and (b) whether there is already concrete, in-session evidence that "passes every automated check" can coexist with "completely broken live" for this exact class of change.** This session shipped two features with the identical "no reachable Chrome" QA gap (`graph-execution-realtime` P2/P3, then `self-coded-graph-migration`) — the first was recorded as a standard partial-QA skip; the second was explicitly flagged as *elevated* risk relative to the first, specifically because (a) it replaced the entire graph-rendering engine rather than adding new UI, and (b) the round-1 review blocker on this exact feature had ALREADY demonstrated, in this same session, that a jsdom-clean/code-review-clean state could ship with a completely-invisible-in-a-real-browser defect. Treating every "QA partially skipped" note as interchangeable would have under-communicated how much riskier this specific ship decision was — the user was shown the differentiated framing before being asked to confirm, not a copy-pasted generic residual-risk paragraph. — **confidence: high** (the differentiation was load-bearing: the user's own answer pattern — "ship, accept the risk" — was the same both times, but the STATE record accurately reflects that the two decisions carried different actual risk weights, which matters for whoever reads this history later trying to prioritize follow-up verification work).

## self-coded-graph-migration — the predicted "no live browser" residual risk materialized within one user session of shipping: SVG default-clips content sized to a static bounding box (2026-07-18)

- **Root cause**: `GraphCanvas.tsx`'s `<svg>` edge layer is explicitly sized via `width`/`height` computed from `content = boundingBox(nodes)` (`graphGeometry.ts`), a `useMemo` keyed only on `nodes` — it has zero dependency on `dragConnect.cursor`. The connect-drag ghost line (a `<path>` from the source node's anchor to the live cursor position) is drawn inside that same fixed-size `<svg>`. SVG elements clip their content to their own dimensions by default (unlike a plain `<div>`, which never clips unless explicitly told to) — so dragging the ghost line past the edge of the existing nodes' bounding box (e.g., toward empty canvas space, exactly as the user's bug report screenshot showed: a dashed line trailing off past the rightmost node into nothing) causes the line to visually vanish mid-drag, clipped by the SVG viewport boundary the box never grows to accommodate. **This is a real regression specific to the self-coded renderer** — xyflow's own canvas never had this failure mode, since it didn't size its SVG root to a content bounding box the same way.
- **How the cause was found**: read `useConnectDrag.ts` first (the natural first suspect for "drag is broken") and found its state machine logically sound in isolation — no bug there. The actual defect was one layer up, in `GraphCanvas.tsx`'s SVG sizing, found by reading the `<svg>` element's `width`/`height` props and asking "what happens if the ghost path's endpoint exceeds these bounds" — a question that doesn't occur to code review focused on interaction logic (state transitions, event handlers) rather than layout/rendering geometry. **Lesson: for a self-coded canvas bug where an element "disappears" mid-interaction, check the containing SVG/viewport's sizing logic before assuming the bug is in the interaction state machine** — a perfectly correct drag-state hook can still produce an invisible result if the geometry it renders into doesn't account for where the interaction can actually go.
- **Direct, concrete confirmation of a prediction made at ship time**: `self-coded-graph-migration-STATE.md` §16 named exactly this risk category when shipping without live QA — "a structurally similar, less-detectable defect could be present in this shipped code right now," drawing on the round-1 review's SVG-namespace bug as precedent that jsdom-clean code can still be broken in a real browser for reasons specific to SVG/DOM content-model edge cases. This bug is a second, independent instance of that exact same risk category (SVG-specific rendering behavior invisible to non-browser verification) surfacing in the same feature, within one user session of shipping — not a coincidence so much as confirmation that the residual-risk framing was correctly calibrated, not overcautious boilerplate. — **confidence: high** (root cause read directly from source, not inferred from the bug report alone — confirmed the `useMemo`'s dependency array excludes `dragConnect`, confirmed SVG's default `overflow` behavior clips, both independently verifiable facts, not speculation).
- **A bug report bundled with an unrelated feature request needs the two separated explicitly, not silently expanded or silently dropped.** The user's report combined "the drag line disappears" (a real, confirmed bug) with "please also allow free node dragging" (a net-new feature reopening a twice-locked `nodesDraggable=false` decision). Investigating and fixing the bug does not implicitly authorize the feature request — asked the user directly which scope was wanted, rather than either bundling the feature into the bugfix (scope creep on a bug ticket) or silently ignoring the second half of the message (dropping a real ask). — **confidence: high** (directly precedented by this session's own repeated pattern of not silently folding adjacent-but-distinct asks into whichever pipeline stage happens to be active).
