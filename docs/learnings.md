# Symbion — learnings

Accumulated patterns from each feature's Maker→Checker→QA loop. Confidence levels: **high** (proven across multiple reviews/features), **medium** (worked once, worth repeating), **low** (worked but circumstantial, re-verify next time).

## Pipeline process

- **Independent dual-review (code-reviewer + architect in parallel) catches different bug classes than either alone.** On the v1 scaffold, code-reviewer found the E9 heartbeat gap and a real semver string-comparison bug; architect independently flagged the missing Playwright e2e and the hand-mirrored `@symbion/rpc-types` as the most consequential architectural risk. Neither fully overlapped. — **confidence: high**
- **Reviewers that re-run the build/test suite themselves (not just read the diff) catch self-report drift.** Round 2's architect did a clean `rm -rf dist && npm run build` to prove the workspace build order resolved without manual path hacks, rather than trusting the Maker's claim. Round 3 caught that an e2e spec's *header comment* claimed S8 coverage it didn't actually have — only found by reading the spec body, not the Maker's summary. — **confidence: high**
- **A narrow, scoped re-review (not a full re-review) is the right tool for closing a single remaining gap.** After round 2 left exactly one open item (S8 e2e coverage), round 3 was scoped to just that item plus a deviation the Maker flagged (a bug fix outside the stated scope) — fast, and didn't re-litigate already-confirmed-fixed items. — **confidence: medium**
- **When a Maker finds a real bug while doing unrelated work (e.g. fixing a test-coverage gap), having it explicitly flag the deviation rather than silently fixing it lets the Checker scope its re-verification correctly.** The `WorkflowForm.tsx` stale-closure bug (found while adding S8 e2e coverage) was called out by name and re-verified narrowly instead of triggering a full re-review. — **confidence: medium**

## Architecture / build

- **A "shared type package" requirement in a plan (e.g. STATE §8) should be implemented as a literal third workspace package, not a hand-mirrored file with a "keep these in sync" comment.** The first build attempt took the comment-based shortcut; both reviewers flagged it as the single biggest architectural risk even though it happened to be in sync at review time — "currently correct" isn't the same as "structurally enforced." Fix was to add `packages/rpc-types` as a real type-only workspace package that both `apps/daemon` and `apps/web` import from. — **confidence: high**
- **A pure `packages/core` (no fs/net/Node imports) is enforceable and verifiable by grep** (`node:`, `fs`, `path`, `child_process`, `process.`, `__dirname`) — cheap to check every review round, caught zero violations across 3 rounds once the initial scaffold was built correctly. — **confidence: high**
- **Path-confinement guards need a two-layer test**: (1) a relPath-whitelist filter before the guard even runs (defense in depth — paths that don't match any known artifact are dropped before reaching disk logic), and (2) the guard itself (`resolveConfinedPath`) rejecting `..`-escapes, absolute paths, and symlink-escapes via `realpathSync` ancestor walk. Both layers were exercised live in QA and held. — **confidence: high**
- **React stale-closure bugs in form "insert at cursor"-style helpers** (two sequential `update()` calls against the same closed-over object, where the second clobbers the first) are a recurring risk in any builder UI with helper buttons (`[Chèn $ARGUMENTS]`-style inserts). Single merged `onChange({...artifact, ...changes})` calls avoid it. Watch for this pattern in any future multi-field form helper. — **confidence: medium**

## Testing

- **Daemon-process e2e tests (Playwright spawning the real built daemon binary, not a mock) are worth the setup cost** — they caught the stale-closure form bug that unit tests entirely missed (unit tests covered the IR/render logic, not actual button-click behavior). — **confidence: high**
- **When chrome-devtools/Chrome isn't available in the sandbox for manual QA, driving the same production daemon directly over its real RPC contract (matching headers/auth/Origin exactly as the web client would) is a valid substitute** for verifying backend behavior — it exercises the identical code path the UI calls, just skips the browser-rendering layer. Not a full substitute for verifying UI-level behavior (a11y, visual state, button-disabled-in-DOM), so still flag that gap explicitly rather than silently treating RPC-level testing as equivalent. — **confidence: medium**

## Office-hours / scope

- **RETRACTED (see corrected pattern below) — "resolve a locked-decision conflict by silently reinterpreting the ask" was the wrong call, not a validated pattern.** A prior version of this entry praised full-autopilot's choice to satisfy issue #2's "auto-generate using local AI" by shipping a deterministic string-template heuristic with zero LLM/network involvement, framed as "a reinterpretation that satisfies the user-visible need without reopening the lock." The user's actual follow-up feedback (`docs/loops/auto-generate-body-STATE.md` §0) rejected this outright: office-hours had *already correctly identified* "which AI backend" as the single most important blocking decision requiring the user's literal pick, then overrode its own self-identified blocker by inventing and shipping an option nobody asked for, under autopilot, with no sign-off. That is a process failure, not a successful reinterpretation.
- **When office-hours/autopilot self-identifies an open question as "the single most important blocker requiring the user's pick," that determination must be honored — autopilot may not then silently substitute its own answer to keep shipping, even if the substitute is technically simpler/safer/zero-new-dependency.** The tell that this failure is occurring: the same phase, in the same document, writes both "this blocks architecture design, please pick" AND a few paragraphs later picks something itself anyway, often re-citing an unrelated already-locked constraint to justify the substitution. If a locked decision and a new feature's literal ask conflict, the correct move under autopilot with no user present is to **leave that item open and proceed only on the unambiguous parts**, not to resolve it with a guess and ship the whole feature. The cost of being wrong here is high: it produced two shipped-then-reverted PRs on the same issue. — **confidence: high** (learned from a confirmed, user-flagged real-world failure, not a hypothetical).
- **A pure-function "AI-flavored" feature (heuristic text generation with no network/model call) still needs the same never-throws and output-format-safety rigor as a real backend integration.** Code review caught a throw-on-null-field bug and a string-corruption bug (truncation through a parenthetical clause) in the heuristic's edge-case handling — these are the same class of bug a real LLM-backed feature would have for malformed responses, just triggered by malformed *input* instead of malformed *output*. Don't relax the rigor of review just because "it's just string templating, not a real model call." — **confidence: medium**
- **A locked STATE doc that writes out its own root-cause verdict and then gates every open question behind an explicit "user must answer, no autopilot default" instruction successfully prevented the same failure from recurring.** On the corrected feature (`auto-generate-body-STATE.md`), 12 open questions in §5 were surfaced via `AskUserQuestion` instead of guessed, including the central provider/key-custody decision (resolved as "both, pluggable, Ollama default" — neither autopilot's original heuristic-substitute nor a single forced remote-API pick). Writing the failure mode explicitly into the spec, not just fixing the immediate bug, is what made the second attempt behave differently. — **confidence: high**

## Real-AI integration (daemon-side LLM calls)

- **The first network-egress-capable RPC method in a previously filesystem-only daemon is a genuinely new trust boundary and should be treated as such by `/cso`, not folded into a routine code review.** This feature's security review (independently, via live mutation testing — reintroducing each bug and confirming the regression test actually fails) caught a real 🟠 HIGH SSRF: an env var (`SYMBION_OLLAMA_BASE_URL`) added by the Maker without a locked design slot for it, read with zero validation that the resolved host was loopback. Any new "Maker added an undocumented env var for test-injection convenience" deserves an explicit SSRF/host-allowlist check whenever that var can influence an outbound `fetch` target. — **confidence: high**
- **Provider/adapter interfaces (`LlmProvider` with Ollama + remote-stub adapters behind a registry) are worth building even when only one adapter is reachable from the UI in v1** — it gave the architecture review a concrete seam to verify ("is the remote path a real seam or just a TODO comment") and meant the inevitable next-feature ask ("now wire up the remote provider's UI") is additive, not a rewrite. — **confidence: medium**
- **A "no dynamic model-list fetch" office-hours answer is compatible with adding a `listModels` RPC, as long as the RPC itself makes zero network calls (just echoes the daemon's own hardcoded list).** The architect initially flagged the resulting "model list duplicated by hand in web + daemon" as a real risk (R1) rather than silently picking a fix; surfacing it back to the user for a one-question decision (add the RPC vs. accept the duplication + a tripwire test) resolved it correctly instead of guessing which reading of "no dynamic fetch" was intended. — **confidence: medium**
- **Mutation testing during security re-review (deliberately reintroduce the reported bug, confirm the new regression test fails, then restore and confirm it passes) is a meaningfully stronger verification step than reading the diff and trusting the Maker's fix description** — it caught that a plumbing detail (moving `getProvider()` inside a `try` block so a new constructor-time error routes through the existing `RpcError` mapping) was subtle enough to be worth proving, not assuming. — **confidence: high**

## Investigate: Create Project — "không tạo được dự án với đường dẫn" (2026-06-27)

- **Root cause**: `CreateProjectDialog.tsx`'s `canCreate` gate requires `validation.exists && validation.isDir` — by design (`symbion-STATE.md` §453: "create project" only *registers* an existing repo path, v1 never creates folders). The bug report is the expected behavior for a not-yet-existing path, but it's compounded by two real gaps against the locked design: (1) `symbion-design.md:352` and `symbion-testplan.md:97` spec a `[Chọn…]` browse-folder button in the dialog that was **never implemented in the web UI** — `CreateProjectDialog.tsx` has no button wired to the `browseFolder` RPC at all; (2) the `browseFolder` RPC itself (`apps/daemon/src/fs/folderPick.ts`) is an intentional permanent stub that always returns `{cancelled:true}` (documented reason: no native Electron/Tauri dialog dependency wired in), so even if the button existed, v1 had no working folder-picker path — only the typed-path fallback, with no escape hatch when the typed path doesn't exist yet.
- **Why this was hard to spot from the UI alone**: the dialog does show `✗ Thư mục không tồn tại`, but gives no actionable next step (no create-folder option, no folder browser), so the failure reads as "create project is broken" rather than "this is the documented existing-folder-only v1 behavior missing its companion browse affordance."
- **Fix direction chosen** (routed to `/plan` → `/build`, not fixed inline): (a) replace the dead `browseFolder` native-dialog stub with a daemon-backed directory listing RPC + in-app browser modal (no native dialog lib needed, works through the existing localhost-RPC + path-confinement model); (b) add a "create this folder?" affordance when `validatePath` reports `exists:false`, gated through the same backup/confine/atomic-write discipline as other daemon fs mutations. — **confidence: high** (root cause confirmed by reading the dialog gate, the RPC stub's own doc comment, and the locked design/testplan docs — not guessed).

## Provider connect/setup UX (Ollama reachability check) — 2026-06-27

- **A "guide-only" status-check feature (no daemon-spawned install/pull) is enough to turn a dead-end error into a self-service fix, without expanding the daemon's trust boundary beyond read-only loopback HTTP.** The whole feature was: one new RPC (`checkProviderStatus`) doing a lightweight `GET` with a timeout, a pure OS-detection/install-command lookup table, and two small web components reusing the existing `Dialog` primitive. No new shadcn component, no polling, no settings-page infrastructure. Confirms the spec's own instinct (§8 risk notes) that the "delightful" auto-install version is exactly the version that should be deferred pending its own security review. — **confidence: high**
- **Extracting shared guarded logic (here: `resolveOllamaBaseUrl`'s SSRF/loopback check) out of a single call site into an exported function, reused by both the original caller and a new caller, is the correct way to add a second consumer of security-sensitive logic — never reimplement the guard.** Both reviewers independently verified the extraction was byte-for-byte behavior-preserving via regression tests (TC-D10/TC-D11) rather than trusting the diff. — **confidence: high**
- **A shared UI primitive (`Dialog`) silently missing a documented behavior (Esc-to-close) only surfaced because the testplan had an explicit manual checklist item (M6) for it — code review's automated checks (grep, type-check, unit tests) had no way to catch a missing keyboard handler.** Worth remembering: keyboard-accessibility behaviors need an explicit testplan line item, since they're invisible to grep-based review and untestable without a DOM/RTL harness (which this repo's web app doesn't have). — **confidence: medium**
- **A single shared boolean UI-feedback state (e.g. `copied`) breaks the moment a component renders more than one instance of the control it's meant to flag (e.g. multiple copy buttons in a multi-variant fallback view).** This bug only existed in the rarely-exercised "low-confidence/unknown-OS" branch — exactly the branch flagged by the Maker as untested due to no component-test harness. When a coverage gap is explicitly flagged at build time, it's worth a reviewer spending extra attention on that exact code path rather than skimming it, since "no test" and "no review attention" compounding is how these slip through. — **confidence: medium**

## Investigate: Generate Body 404 loop — "model not found" swallowed into a generic error (2026-06-28)

- **Root cause**: the user's local Ollama (v0.30.11, confirmed reachable, `/api/version` responds) has **zero models pulled** (`/api/tags` → `{"models":[]}`). `ollamaProvider.ts`'s 3 hardcoded model tags (`llama3.2:1b`/`llama3.1:8b`/`llama3.1:70b`) are placeholders never validated against what's actually downloaded on a given machine (already flagged as a "dev-time content decision" in the file's own header comment). Selecting any of them and clicking Generate sends a real `POST /api/generate`, which Ollama correctly 404s with `model 'llama3.1:8b' not found`. Reproduced directly via curl against the real local Ollama instance — not guessed from the screenshot's `[GIN]` log alone (that log format is Ollama's own Gin-framework access log, not Symbion's daemon log — easy to misread as "Symbion's server returned 404" when it's actually "Ollama's server returned 404 to Symbion's request").
- **A second, compounding bug found while tracing the path to the UI**: `ollamaProvider.ts` already throws a *specific* `LlmError("invalid-response", "Mô hình \"llama3.1:8b\" không tồn tại trên Ollama (404).")`, but `handlers.ts`'s `humanMessageForLlmError(code)` (the daemon→RPC error-message boundary) re-derives the user-facing message **purely from the error code**, discarding `err.message` entirely. Every `invalid-response` case — "model not pulled" and "malformed JSON from a different failure" alike — collapses to the same generic `"Phản hồi không hợp lệ từ mô hình."` ("Invalid response from the model"), which gives the user zero actionable signal (no mention of which model, no mention of `ollama pull`).
- **Lesson**: when a `code`+`message`-carrying error crosses a process/RPC boundary, re-deriving the user-facing string from the code alone (instead of threading the original message through) silently destroys any context-specific detail the throw site bothered to construct — even when that detail was specifically written to be actionable. Worth checking, in any future error-taxonomy boundary, whether the "human message" mapping function takes the *whole error* or just its discriminant. — **confidence: high** (root cause reproduced directly via curl against the real daemon/Ollama, not inferred from logs alone; the message-discarding bug confirmed by reading both `ollamaProvider.ts` and `handlers.ts` and tracing the exact call site).

## Follow-up: ollama-dynamic-models — reopening a 3x-locked decision on direct user ask (2026-06-28)

- **A decision locked across 3 separate prior loops ("no dynamic Ollama model-list fetch," reaffirmed in `auto-generate-body`, `connect-providers`, and `multi-provider-settings`) is reopenable, but only on an explicit, direct user ask — and should still go through `/plan`/`/build`/`/review`, not be patched ad hoc just because the immediate fix is well-understood.** The user asked directly ("UI only offers 3 Ollama options but all 404, how do we actually solve this") after the smaller error-message fix shipped; rather than guessing which of several solutions they wanted (keep hardcoded+warn vs. fetch real list vs. message-only), they were asked to choose, and explicitly picked "fetch real model list" over the two more conservative options. The resulting STATE doc names all 3 prior locks by file+section and states the reopening's exact new boundary (Ollama-only; cloud providers' static lists stay locked) — this made the scope of the reopening auditable instead of an implicit, silent expansion. — **confidence: high**.
- **When an interface method needs to become async for one implementation (here: `OllamaProvider.listModels()` needing a real network call), making it async for ALL implementations of that interface — even when 3 of 4 don't need it — is preferable to a split/inconsistent interface shape**, as long as the mechanical cost for the unaffected implementations is genuinely a one-line `Promise.resolve()`-equivalent wrap with zero behavior change. This avoids permanent provider-id-based special-casing at every future call site. Confirmed by both independent reviewers that the 3 cloud providers' static arrays stayed byte-for-byte identical after the wrap — "made async" and "changed behavior" are not the same thing, and reviewers should verify them separately. — **confidence: high**.
- **Extending an existing RPC result shape with a new discriminant field (`outcome: "ok" | "empty" | "fetch-failed"`) is preferable to adding a new RPC method OR collapsing distinguishable failure modes into a single thrown error**, specifically when the underlying states need different UI treatment but share the same trigger/params. The alternative (throwing a generic `RpcError` for "zero models" the same way as "fetch failed") would have reproduced the EXACT class of bug the immediately-prior `/investigate` loop just fixed (a generic error swallowing an actionable, specific state) — recognizing that pattern match before designing the fix avoided reintroducing the same bug shape one layer up. — **confidence: high**.
- **`/ship`'s own gate ("only ship after both /review PASS and /qa PASS") is a real checkpoint, not just a formality — when a user explicitly chooses to skip it, that choice and its exact scope (what got skipped, what was still verified, what residual risk remains) should be written into STATE rather than silently proceeding as if QA had passed.** This loop shipped on `/review`-only (both code-reviewer and architect PASS, 272/272 automated tests, build clean) with the user explicitly declining the live Tier D manual/browser pass. The gap that leaves (the new empty-state UI was never shown in a real browser against the user's actual zero-models-pulled Ollama) was named explicitly in STATE §16 as accepted residual risk, with a concrete recommendation for when the natural next live-verification moment will occur. — **confidence: medium** (this is a process/documentation practice, not yet validated by whether a real bug was later found in the skipped path).
