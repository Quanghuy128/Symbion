## 1. Original request (translated/condensed)

Symbion currently renders its dependency graph with React Flow v11 (`apps/web/src/components/DependencyGraph.tsx` + `apps/web/src/components/graph/*.tsx`, ~1800 lines). This is a **fixed auto-layout dependency map** (`nodesDraggable={false}` — not a free-form drag-drop executor canvas), with:
- custom node types (`command`, `agent`, `missingAgent`)
- custom animated bezier edges with count/goal decoration (`AnimatedEdge`)
- connect-by-drag (command → agent) with validation (`isValidConnection`, `onConnect`)
- hover highlight/dim, click-to-select edge (pinned +/× toolbar)
- pane context menu (add node, fit view)
- mission-mode run overlay (glow/dim participant nodes+edges) — shipped 2026-07-15 in `graph-execution-realtime` P1

License is **not** the driver this time (React Flow core is MIT/free — already confirmed in `docs/loops/graph-execution-realtime-STATE.md` §2 two days ago). The driver is: **is there a lib or hand-rolled approach with better long-term flexibility and UX/DX for this specific use case** (fixed layout, heavy custom decoration, growing interactivity)?

Requested comparison set, each scored against explicit criteria:

| Candidate | Notes |
|---|---|
| React Flow (baseline) | current, ~1800 LOC of custom node/edge/interaction code already built on top |
| Self-coded SVG/Canvas | optionally paired with `dagre` or `elkjs` for auto-layout only (not interaction) |
| 1–2 lighter alternatives | e.g. `reaflow`, or `@xyflow/react`'s lower-level primitives (same package, different API surface) |

Evaluation axes requested by the user:
1. **Port effort** — re-implementing every existing feature (custom nodes/edges, connect validation, run-engine glow overlay, hover/select, context menu) on the candidate.
2. **Bundle size** impact.
3. **Future UI/UX customizability** — e.g. enabling node dragging later if wanted.
4. **Migration risk** on a surface under **active development** (run-engine P2/P3 follow-ups already scoped in `graph-execution-realtime-STATE.md`).

## 2. Core user need / motivation (the "why")

The underlying motivation is **not** a broken feature or a performance complaint today — it is a **preemptive DX/architecture question**: "is the current library still the right foundation before we build more on top of it (P2 telemetry roll-up, P3 history/reattach, possibly future node-dragging/free layout)?" The cost of being wrong compounds — every sprint spent building more custom decoration atop React Flow increases future port cost if a switch is ever justified. The user wants a **researched, criteria-scored decision**, not an opinion, and wants it **before** committing more surface area to React Flow's API shape.

This is fundamentally a **build-vs-keep** / technical-debt-avoidance decision, informed by the fact that a closely related decision (license) was already made and documented 2 days ago in a sibling feature — this spike is a **narrower, deeper second pass** on the same question, now optimizing for flexibility/DX rather than cost/license.

## 4. Constraints (bring forward to architect if migration is recommended)

- **Filesystem safety / no-silent-write invariants are irrelevant to this UI-only surface** (the graph reads `artifacts` from a store; it never writes files directly — all mutations funnel through `saveArtifact`/`deleteArtifact` to daemon RPC). A library swap must **preserve** this: the new graph library must remain presentation-only, deriving nodes/edges from `artifacts` (E10 invariant, see `DependencyGraphInner`'s own doc comment: "Nodes/edges stay DERIVED from `artifacts`... never mirrored into local nodes/edges state").
- **Active-development surface**: `graph-execution-realtime` P2 (token roll-up UI) and P3 (history/reattach/settings) are already scoped as follow-ups on top of the *current* React Flow node/edge data-bag shape (`CommandNodeData`, `AnimatedEdgeData`). A migration recommendation must state whether it should happen **before** P2/P3 (clean cut) or **after** (avoid rewriting twice) — this is a sequencing question the answer must address, not skip.
- **Provider-format fidelity / IR purity** — not applicable here (graph lib choice is `apps/web`-only, never touches `packages/core`).
- Must stay inside `apps/web` — no new backend/daemon RPC implied by a graph-lib swap.

## 5. Acceptance criteria — feature-preservation checklist (what "complete port" means, for scoring candidates)

Any candidate is scored as "full parity," "partial parity (list gaps)," or "no reasonable path," against this concrete, testable checklist derived from the current code:

- **AC-1** Distinct visual node types: `command` (indigo box, source handle only), `agent` (target handle only), `missingAgent` (dashed/warning, "create agent" affordance) — verified against `CommandNode.tsx`/`AgentNode.tsx`/`MissingAgentNode.tsx`.
- **AC-2** Custom bezier edges with: staggered draw-in animation (capped at 15 edges, 40ms stagger), dashed style for "missing" edges, a decoration badge (×N count pill or goal dot) — verified against `AnimatedEdge.tsx`.
- **AC-3** Edge interactivity: wide invisible hover hit-area (20px) revealing a floating +/× toolbar (edit-relationship modal / unlink-with-inline-confirm), click-to-pin the toolbar open (`selectedEdgeId`), and a distinct "pending ghost" edge state (dashed + spinner) shown only locally during an in-flight save, never persisted to any node/edge store.
- **AC-4** Connect-by-drag from a `command` handle to an `agent` handle, with **live validation** (`isValidConnection`) rejecting self-loops and non-command→agent pairs, plus a duplicate-mention no-op guard.
- **AC-5** Hover-driven highlight/dim across the whole graph (hovered node + its direct edges/neighbors stay full-opacity; everything else dims to 35%), suspended during mission mode.
- **AC-6** Pane-level context menu (right-click empty canvas) offering add-node actions + fit-view, positioned at cursor.
- **AC-7** "Just added" transient ring (accent outline that auto-fades ~1.6s) on any newly created node, detected via id-set diffing against the previous render.
- **AC-8** Mission-mode run overlay (from `graph-execution-realtime` P1, shipped same day): participant nodes/edges stay full-opacity + a glow ring (color keyed to `runStatus`: active/done/error/cancelled) while non-participants dim to 35%; edges expose a `runFlow` "flowing"/"off" state; **all authoring interactivity (drag-connect, edge +/× toolbar, context menu, delete) is suspended while a run is active** (`authoringSuspended`).
- **AC-9** Auto-layout: nodes get computed `(x, y)` positions with no manual dragging (`nodesDraggable={false}`); `fitView` on demand and on mount.
- **AC-10** Perf/DX floor: candidate does not regress the "derive nodes/edges via `useMemo` off `artifacts`, never mirror into a separate mutable node/edge state" architecture (E10) — i.e., whatever primitive is chosen must support pure-functional re-render from external state without requiring the app to hand-manage a parallel node/edge store.
- **AC-11** Deliverable is a written comparison (recommendation doc, not code) covering, per candidate: (a) itemized AC-1..AC-10 parity/gap, (b) estimated port effort (rough size class: small/medium/large, not a time estimate — architect can size story points), (c) bundle-size delta (approximate, e.g. via published package sizes/bundlephobia-style data), (d) customizability outlook for hypothetical future node-dragging, (e) migration-risk callout given P2/P3 are actively being layered on the current implementation.
- **Definition of done for this spike**: a decision recorded in this STATE file (or a follow-on doc) that is one of **"keep React Flow"** / **"migrate to X"** / **"revisit after P2/P3 ship"** — with the comparison table backing it up, and, if migration is chosen, a clear statement of *when* (now vs. after P2/P3) for architect to plan around.

## 6. Open questions — RESOLVED (user decision, 2026-07-15)

1. **Sequencing**: **after** P2 (token roll-up UI) and P3 (history/reattach) ship and stabilize. Do not start a migration while the node/edge data-bag shape is still moving — avoid writing custom-node code twice. This is a **hard precondition** for any migration work, not just a preference.
2. **"Lighter alternative" candidates**: delegated to architect/dev — pick 1-2 during the research pass (desk research, not a user pre-lock). Candidates already surfaced in the sibling `graph-execution-realtime` research (Cytoscape.js, Sigma.js, AntV G6, vis-network) plus `reaflow` are fair game.
3. **Depth of the self-coded SVG/Canvas estimate**: desk-estimate (reading source, reasoning about effort) is sufficient for this pass — **no throwaway prototype**, since the outcome may well be "not worth switching" and a prototype spends real engineering time either way. Revisit if the desk-estimate can't produce a trustworthy port-effort number.
4. **Trigger/urgency — CONFIRMED, there is a concrete pain point** (not pure due-diligence): the user named three concrete frictions with the current React Flow integration:
   - **Hard to customize UI/style** — custom node/edge visuals feel constrained by React Flow's API shape (e.g. `NodeProps`/`EdgeProps`, `Handle`, `EdgeLabelRenderer` positioning) rather than free-form.
   - **Performance/lag** with a larger graph or frequent re-renders (mission-mode's live glow/dim updates are the likely aggravating case — every run-status tick re-derives `nodes`/`edges` via `useMemo`, see `DependencyGraph.tsx:444-504`).
   - **API/DX friction when adding new features** — mission-mode itself (`f65b34b`) was cited as an example of this being harder than it should be, even though §7 below notes it technically only touched `data`/style, not new React Flow APIs — this suggests the friction is more about *ergonomics of extending the existing custom-node contract* than about hitting a hard React Flow limitation. The research should verify which it actually is.
   This sharpens AC-11d (customizability outlook) beyond the hypothetical "node dragging": score each candidate explicitly against (a) style/visual customization freedom, (b) re-render/perf cost under mission-mode-style live updates, (c) ergonomics of adding a new node/edge data field or interaction in the future.
5. **Consumer of the output**: gates a real decision — if the research recommends migrating, that becomes a future feature (own `/analyze`→`/build` pipeline, gated on P2/P3 completion per #1). Not just a `docs/learnings.md` note.

## 8. Ideas beyond the literal ask (product angle)

The three concrete "flexibility/DX" wants implied by the request are **separable, independently-shippable decisions** that should not stay bundled behind "replace React Flow" — each has a different risk/cost profile:

1. **Upgrade `reactflow@^11` → `@xyflow/react` v12** (same team/license, v11 is maintenance-mode) — packaging/perf only, near-zero UX risk, no AC-1..AC-10 rewrite needed.
2. **Improve auto-layout** with `elkjs`/`dagre` (layered/mrtree algorithm) instead of today's naive `{x: 0/320, y: i*80}` two-column stack — pure algorithm swap, zero node/edge-type risk, likely higher perceived value per dev-hour than a rendering-library migration.
3. **Node dragging + persisted manual layout** (`nodesDraggable=true` + saved `{x,y}`) — a product/data-model decision (where does layout live: IR field vs. `.symbion/` sidecar, since core IR is meant to stay vendor-agnostic/presentation-free), not a rendering-library decision.

Also unaddressed by either design doc and worth a decision either way, independent of the library question: a **MiniMap/Controls** for larger graphs (near-zero cost, stock React Flow subcomponents), and **touch/tablet support** (today's connect-drag, hover-reveal toolbars, and hover midpoint menus are all mouse-first — undocumented as desktop-only or in need of a fallback).

**Recommendation woven into open question #2**: evaluate/ship (1) and (2) now as small, low-risk changes regardless of what this spike concludes; keep (3) and any full-library-swap decision gated on this spike's output.

### 10.0 Flaws found in the STATE spec itself (flagging per architect mandate, not silently absorbing)

Before scoping execution, two things in §1-§9 deserve pushback rather than blind implementation:

1. **§6.2 "lighter alternative" candidate list is stale/overbroad relative to §6.4's sharpened axes.** Cytoscape.js, Sigma.js, AntV G6, vis-network were surfaced in the *prior* spike (graph-execution-realtime, license/cost framing) — none of them were evaluated there for "style/visual customization freedom" or "ergonomics of extending the node/edge contract," because that wasn't the question being asked then. Carrying that same candidate list forward risks looking like due diligence while actually just re-scoring old candidates against new axes without re-checking whether they're even good fits. **Decision for this plan**: narrow to candidates that are plausible on the *new* axes specifically — see §10.1 below for the reasoning and the final pick.
2. **§5 AC-11 asks for "bundle-size delta" but §6.3 forbids a prototype** — these are in mild tension. A desk-estimate bundle-size number (published npm/bundlephobia figures) is directionally fine for a go/no-go decision but is NOT precise (tree-shaking, peer-dep overlap with React/React-DOM already in the app, gzip vs. brotli reporting differences across sources). **This plan requires the deliverable to caveat every bundle number as "desk-estimate, ±30%,"** not present false precision — a table with bare KB numbers and no caveat would be a research-quality defect, not just a formatting nit.
3. **The three pain-point axes in §6.4 are subjective/qualitative by nature** ("hard to customize," "feels constrained") — there is a real risk the research produces vibes dressed as a scorecard. This plan requires each pain-point score to cite a **specific, concrete code location or API touchpoint** (e.g. "`EdgeLabelRenderer` requires portal-absolute positioning, see `AnimatedEdge.tsx:140-160`, vs. candidate X's declarative label API") rather than a bare adjective — this is the single biggest lever against an untrustworthy conclusion (see Edge Cases §10.3).

### 10.1 Candidate selection (delegated by STATE §6.2 — decision made here)

Given §6.4's three sharpened axes (style/visual customization freedom, re-render/perf cost under live updates, ergonomics of extending node/edge contract) rather than the old cost/license framing, the two candidates chosen for deep comparison are:

- **Cytoscape.js** — carried forward from the prior spike's candidate list because it is explicitly a graph-first (not React-first) library with a headless style/layout engine (its own CSS-like stylesheet DSL) — genuinely different on the "customization freedom" axis, worth re-scoring under the new question even though it was rejected once already (that prior rejection was on a *different* axis — see §10.4 precedent-engagement requirement).
- **`@xyflow/react` v12 lower-level primitives** (same package as today, different API surface: `useNodesState`/`useEdgesState` bypassed in favor of raw `<Handle>`/custom pane composition, or the "controlled" pattern without the wrapper conveniences) — the STATE text explicitly names this as a fair candidate (§1 row 3, "same package, different API surface"). This is the cheapest possible experiment to reason about (zero new dependency, all existing custom-node/edge code is untouched, only the state-management wrapper changes) and directly tests the hypothesis raised in §6.4 #4: is the friction a hard React Flow ceiling, or ergonomics of *our* current wrapper usage?

**Explicitly rejected for deep comparison** (documented, not silently dropped, to close the loop the prior spike opened):
- Sigma.js — WebGL/huge-graph oriented; Symbion's graphs are ~10-30 nodes; wrong tool class for either axis set, same conclusion as the prior spike, no new information changes this.
- AntV G6 — closest "canvas-first, still configurable" alternative to Cytoscape but weaker English-language docs/community than Cytoscape for troubleshooting; one candidate in this bucket is enough to test the "canvas/non-React-first" hypothesis, Cytoscape is the more defensible pick.
- vis-network — smaller community, less active maintenance cadence than the others; would need a maintenance-health check before even desk-research (see Edge Case E-2) — not worth spending the 1-2 candidate budget on.
- `reaflow` (named as an example in §1 row 3) — checked for maintenance health first (see §10.2 step 0); if it fails that gate it is dropped without a deep dive, since a candidate recommendation resting on a barely-maintained library would itself be an untrustworthy conclusion (Edge Case E-2).
- Self-coded SVG/Canvas is NOT "rejected" — it is the third leg of the required comparison per §1/§3, given its own estimation method in §10.2 step 4.

### 10.4 Edge cases (things that make the research spike itself untrustworthy)

- **E-1 Stale/vendor-biased bundle-size or benchmark claims.** A blog post or vendor doc citing "X is 3x faster than React Flow" may be old (pre-v12 React Flow, pre-major-version-bump candidate), synthetic-benchmark-only (not representative of Symbion's ~10-30 node graphs), or vendor-marketing copy for the candidate. Mitigation: prefer primary sources (the library's own CHANGELOG/release notes, actual npm package size) over secondary blog claims; date every cited source; if a perf claim can't be dated or sourced to primary docs, mark it "unverified, directional only" rather than citing it as fact.
- **E-2 A "lighter" candidate turns out unmaintained.** Mitigation: Step 0's explicit maintenance-health gate (npm last-publish, GitHub last-commit, archived-repo check) BEFORE deep scoring — this is a design-level defense, not an afterthought, precisely because it's cheap to check and expensive to discover after writing a full scorecard for a dead library.
- **E-3 Conflating "hard to customize" with "haven't tried the documented extension point yet."** This is directly named in STATE §6.4 #4. Mitigation: Step 2 requires reading the ACTUAL mission-mode diff before asserting the friction was a React Flow ceiling vs. an ergonomics-of-our-own-wrapper issue — an assertion here without that citation is a red flag a reviewer should catch (see testplan §T-3).
- **E-4 Prototype creep.** STATE §6.3 forbids a throwaway prototype. Mitigation: if during desk research it becomes clear a claim (e.g. bundle size, perf) genuinely cannot be estimated without running code, the correct move is to say so explicitly in the deliverable ("desk-estimate cannot resolve X; would require a timeboxed spike to verify") rather than either (a) quietly writing a prototype anyway, violating the locked constraint, or (b) inventing a number. This escape hatch must be named, not silently exercised.
- **E-5 Precedent contradiction without acknowledgment.** Already covered as its own mandatory Step 5 above; listed here too because it's the single highest-value check a reviewer should perform (STATE §7's explicit warning about "conflicting decisions... process drift").
- **E-6 Scope creep into a migration architecture.** STATE §3 explicitly reserves migration architecture for a *later*, separate `/plan` if "migrate" is recommended (see §7: "do not let `/plan` skip straight to architecture on the strength of this spec alone"). - **E-7 Treating the three pain-point axes as if they were objectively measurable when they're partly a taste/preference call.** Mitigation: require citations (§10.0 flaw #3) to keep the scoring as evidence-grounded as the subject matter allows, but the deliverable should say plainly where a judgment call remains a judgment call rather than dressing it up as more rigorous than it is.
